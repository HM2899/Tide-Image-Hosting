# 潮汐图床代码规范与交付标准

## 1. 文档目的

本文档用于约束潮汐图床后续所有开发工作，确保项目以企业级工程标准推进。

所有功能开发必须严格按照现有需求文档、数据库设计、接口文档、前端页面清单、存储适配器设计和开发排期执行。开发过程中不得擅自增加、删除、改变需求，不得绕过既定架构，不得以临时实现替代正式方案。

## 2. 最高原则

- 一切需求以 `docs/` 目录中的项目文档为准。
- 未写入文档的功能不得擅自实现。
- 需求不明确时，必须先补充文档或向项目负责人确认。
- 每完成一个功能，必须自行测试。
- 每完成一个功能，必须自行核查问题。
- 每次交付前，必须按企业项目标准进行整体校验。
- 不允许只实现表面功能而跳过权限、错误处理、测试和边界场景。
- 不允许为了赶进度破坏架构设计。
- 不允许在代码中编写注释。

## 3. 需求执行规范

### 3.1 开发前检查

开始任何功能前，必须确认：

- 该功能是否已写入需求文档。
- 该功能是否涉及数据库变更。
- 该功能是否涉及接口变更。
- 该功能是否涉及前端页面。
- 该功能是否涉及权限控制。
- 该功能是否涉及存储器。
- 该功能是否涉及审核、配额、回收站、日志或备份。
- 该功能是否需要测试用例。

如果发现文档缺失，先补文档，再写代码。

### 3.2 不得擅作主张

禁止行为：

- 擅自改变接口路径。
- 擅自改变数据库字段含义。
- 擅自绕过用户组配额。
- 擅自绕过审核流程。
- 擅自绕过存储适配器直接操作存储。
- 擅自改变前端交互逻辑。
- 擅自新增未确认的第三方依赖。
- 擅自删除已规划功能。
- 擅自将第一版功能推迟到后续版本。
- 擅自修改项目定位。

允许行为：

- 修复明显错误。
- 补充缺失的测试。
- 按文档细化实现细节。
- 在不改变需求的前提下优化代码结构。
- 在发现冲突时暂停并提出问题。

## 4. 代码风格规范

### 4.1 通用规范

- 代码必须清晰、简洁、稳定。
- 命名必须准确表达业务含义。
- 避免过度抽象。
- 避免重复实现。
- 避免魔法值。
- 避免超长函数。
- 避免超大文件。
- 避免隐藏副作用。
- 避免把业务逻辑写在前端。
- 避免把权限判断分散在多个无关位置。
- 禁止提交调试代码。
- 禁止提交临时打印。
- 禁止提交无用文件。
- 禁止提交未使用变量、未使用函数、未使用依赖。

### 4.2 注释规范

项目代码中不写注释。

禁止：

- 行内注释。
- 块注释。
- 函数说明注释。
- 临时 TODO 注释。
- 用注释解释混乱代码。

要求：

- 用清晰命名表达含义。
- 用合理拆分表达流程。
- 用类型表达约束。
- 用测试表达行为。
- 用文档表达业务规则。

如果出现必须解释的复杂逻辑，优先重构代码结构；如果业务规则必须说明，写入 `docs/` 文档，而不是写在代码注释中。

### 4.3 Rust 代码规范

Rust 后端代码必须遵守：

- 使用 `cargo fmt` 格式化。
- 使用 `cargo clippy` 检查。
- 禁止 `unwrap` 处理业务错误。
- 禁止 `expect` 处理可恢复错误。
- 禁止吞掉错误。
- 错误必须转换为统一业务错误。
- 数据库访问必须走 repository 或 service 分层。
- HTTP handler 不直接写复杂业务逻辑。
- 存储操作必须走 StorageProvider。
- 权限校验必须在统一位置执行。
- 密码必须使用安全哈希。
- 敏感配置必须加密保存。
- 所有外部服务调用必须设置超时。
- 所有外部服务调用必须处理失败。

推荐分层：

```text
routes
handlers
services
repositories
models
dto
auth
storage
audit
settings
tasks
errors
```

### 4.4 前端代码规范

前端必须遵守：

- 页面结构清晰。
- 组件职责单一。
- 表单校验完整。
- 错误提示明确。
- 加载状态明确。
- 空状态明确。
- 移动端必须可用。
- 交互必须符合前端页面清单。
- 不允许只实现桌面端。
- 不允许把管理员能力暴露给普通用户。
- 不允许在前端硬编码敏感配置。
- 登录、注册、重置密码不得作为首页同级主卡片，必须以弹窗、抽屉或覆盖层等辅助入口呈现。
- 未登录访问私有页面时必须显示登录引导，不得请求私有接口。
- 上传失败、登录失败和接口失败不得直接展示完整 JSON、HTML 或原始响应体。
- 公共图库只允许展示正常公开图片。
- 主题外观必须由后台配置驱动，至少支持圆角、模糊强度、透明度和配色。
- 管理员后台一级菜单必须保持为仪表盘、图片与标签、用户与配额、审核管理、存储与迁移、系统设置 6 项。
- 上传、随机图、外观、日志、备份恢复必须收纳到系统设置，不得重新拆成一级菜单。
- 标签能力必须保留在图片与标签中，不得作为单独一级菜单。

所有图片列表默认使用 WebP 预览图。

所有危险操作必须二次确认：

- 删除图片。
- 永久删除图片。
- 封禁用户。
- 删除用户。
- 删除存储配置。
- 启动迁移。
- 恢复备份。

### 4.6 前端交付验收

前端交付前必须执行：

- 验证错误密码登录会显示中文短提示。
- 验证管理员可使用用户名或邮箱从普通登录入口登录。
- 验证正确登录后 `GET /api/auth/me` 可刷新当前用户。
- 验证管理员登录后侧边栏显示管理员后台入口，普通用户和未登录用户不可见。
- 验证登录用户可上传图片。
- 验证访客上传关闭时显示中文短提示。
- 验证公共图库未登录可访问且只显示公开正常图片。
- 验证未登录打开我的图片、回收站、个人资料、API Token 时只显示登录引导。
- 验证未登录导航只展示首页上传和公共图库，不展示我的图片、回收站、我的、API Token、图片详情入口。
- 验证桌面端不存在顶栏和侧边栏重复主导航。
- 验证管理员后台只展示 6 个一级入口。
- 验证标签管理位于图片与标签二级入口。
- 验证上传、随机图、外观、日志、备份恢复位于系统设置二级入口。
- 验证移动端后台使用抽屉菜单，窄屏后台表格能转为卡片式列表。
- 验证公共图库、我的图片和上传结果点击图片可打开预览弹窗。
- 验证 Markdown 链接入口打开部署引用弹窗，且包含原图 URL、WebP URL、原图 Markdown、WebP Markdown、原图 HTML、WebP HTML。
- 验证历史 `localhost:8080`、`127.0.0.1:8080`、`0.0.0.0:8080` 本地存储链接会按当前 `PUBLIC_BASE_URL` 归一化。
- 验证上传空文件、非图片文件或访客上传关闭时显示中文短提示，并且不会持续触发失败上传请求。
- 验证后台外观设置保存后前端样式刷新生效。
- 重新构建 `frontend/` 静态产物。
- 截取桌面和移动端验收截图。

### 4.5 SQL 与数据库规范

- 所有数据库结构变更必须写 migration。
- 禁止直接手动改库作为正式方案。
- 字段命名使用 snake_case。
- 表名使用复数形式。
- 重要查询必须考虑索引。
- 删除操作优先软删除。
- 敏感配置必须加密。
- JSON 字段只用于配置、扩展信息和第三方返回结果。
- 核心业务字段不得全部塞进 JSON。

需要重点索引：

- `users.email`
- `images.user_id`
- `images.file_object_id`
- `images.status`
- `images.created_at`
- `file_objects.sha256`
- `storage_objects.file_object_id`
- `storage_objects.storage_provider_id`
- `tags.slug`
- `image_tags.image_id`
- `image_tags.tag_id`
- `quota_usage.user_id`
- `quota_usage.date`

## 5. 架构边界规范

### 5.1 存储边界

所有图片相关存储操作必须走存储适配器。

禁止：

- 业务层直接拼接 R2 URL。
- 业务层直接调用 OneDrive API。
- 业务层直接调用 Oracle API。
- 前端决定图片真实存储位置。
- 随机图接口绕过 StorageProvider。

必须通过 StorageProvider 处理：

- 原图上传。
- WebP 预览图上传。
- 头像上传。
- 图片删除。
- 图片访问 URL。
- 签名 URL。
- 系统代理 URL。
- 存储健康检查。
- 图片迁移。

### 5.2 审核边界

所有审核逻辑必须走审核服务。

上传审核采用后置执行：上传接口必须创建审核任务，图片先以正常状态展示；审核拒绝后必须通过统一删除服务永久删除图片和存储对象，并保留审核结果与日志。

禁止：

- 上传接口直接写死审核结果。
- 访客上传绕过审核。
- 普通用户绕过用户组审核策略。
- AI 审核失败后直接默默放行。

审核失败策略必须来自后台配置。

### 5.3 配额边界

所有上传、API 调用、随机图调用必须检查配额。

禁止：

- 只在前端显示配额，不在后端校验。
- 访客上传不计入限制。
- API 上传绕过配额。
- 批量上传只检查总请求，不检查单文件。

### 5.4 权限边界

权限必须后端强校验。

普通用户只能：

- 查看自己的图片。
- 删除自己的图片。
- 恢复自己的图片。
- 修改自己的资料。
- 管理自己的标签关联。

管理员可以：

- 管理用户。
- 管理全站图片。
- 管理审核。
- 管理存储。
- 管理系统配置。

超级管理员额外可以：

- 管理管理员。
- 修改关键安全配置。
- 执行恢复备份等高危操作。

## 6. 错误处理规范

所有错误必须明确分类：

- 参数错误。
- 认证错误。
- 权限错误。
- 配额不足。
- 文件类型不允许。
- 文件过大。
- 审核拒绝。
- 存储错误。
- 数据库错误。
- 外部服务错误。
- 系统内部错误。

接口错误必须返回稳定错误码。

示例：

```text
AUTH_REQUIRED
PERMISSION_DENIED
QUOTA_EXCEEDED
FILE_TOO_LARGE
FILE_TYPE_NOT_ALLOWED
IMAGE_REJECTED
STORAGE_UNAVAILABLE
DATABASE_ERROR
EXTERNAL_SERVICE_ERROR
INTERNAL_ERROR
```

错误信息要求：

- 用户可见错误要清楚。
- 管理员日志要详细。
- 不向普通用户暴露敏感内部信息。
- 不暴露密钥、Token、私钥、数据库连接信息。

## 7. 安全规范

### 7.1 密码与认证

- 密码必须使用 Argon2。
- Session 必须有过期时间。
- API Token 只保存哈希。
- 邮箱验证码必须有过期时间。
- 登录失败必须限流。
- 管理员高危操作必须二次确认。

### 7.2 文件安全

- 上传必须检查扩展名。
- 上传必须检查 MIME。
- 上传必须进行图片解码验证。
- 禁止危险文件作为公开图片直出。
- EXIF 处理必须可配置。
- 访客上传必须支持验证码。

### 7.3 配置安全

必须加密保存：

- SMTP 密码。
- OneDrive 客户端密钥。
- R2 Secret Access Key。
- Oracle Secret Access Key。
- Oracle Private Key。
- 大模型 API Key。
- FastAPI Token。

### 7.4 日志安全

日志中禁止记录：

- 明文密码。
- 明文 Token。
- 明文密钥。
- 明文私钥。
- 完整验证码。

## 8. 测试规范

### 8.1 每个功能必须测试

每写完一个功能，必须至少完成：

- 正常路径测试。
- 权限测试。
- 参数错误测试。
- 边界值测试。
- 失败场景测试。
- 回归影响检查。

### 8.2 后端测试

后端测试覆盖：

- service 单元测试。
- repository 数据库测试。
- handler 接口测试。
- 权限测试。
- 配额测试。
- 存储适配器测试。
- 审核策略测试。
- 迁移任务测试。
- 备份恢复测试。

### 8.3 前端测试

前端测试覆盖：

- 页面加载。
- 表单校验。
- 登录态。
- 权限态。
- 空状态。
- 加载状态。
- 错误状态。
- 移动端布局。
- 关键按钮操作。

### 8.4 存储测试

每个存储适配器必须测试：

- 测试连接。
- 上传原图。
- 上传 WebP 预览图。
- 读取元数据。
- 生成访问 URL。
- 生成签名 URL。
- 删除对象。
- 失败重试。

覆盖存储：

- 本地存储。
- Cloudflare R2。
- OneDrive。
- Oracle S3 兼容模式。
- Oracle OCI 原生模式。

### 8.5 上传链路测试

必须测试：

- 登录用户上传。
- 访客上传。
- 超过大小限制。
- 不允许的文件类型。
- 哈希重复文件。
- WebP 预览图生成。
- 需要审核。
- 审核通过。
- 审核拒绝。
- 删除到回收站。
- 恢复图片。
- 永久删除图片。

### 8.6 管理员后台测试

必须测试：

- 普通用户不能访问后台。
- 管理员可以访问后台。
- 管理员可以封禁用户。
- 管理员可以删除图片。
- 管理员可以审核图片。
- 管理员可以修改配额。
- 管理员可以添加存储。
- 管理员可以测试存储。
- 管理员可以启动迁移。
- 管理员可以创建备份。

## 9. 自检流程

每完成一个功能，开发者必须按以下顺序自检：

1. 对照需求文档确认功能范围。
2. 对照数据库设计确认字段和关系。
3. 对照接口文档确认路径、参数、返回值。
4. 对照前端页面清单确认页面状态。
5. 对照存储适配器设计确认没有绕过存储层。
6. 执行格式化。
7. 执行静态检查。
8. 执行相关测试。
9. 手动验证核心流程。
10. 检查日志是否泄露敏感信息。
11. 检查权限是否正确。
12. 检查移动端是否可用。
13. 记录本次完成内容和遗留问题。

## 10. 交付前企业级校验

每次交付给项目负责人前，必须完成以下校验。

### 10.1 工程校验

- 项目可以正常构建。
- 项目可以正常启动。
- 数据库 migration 可从空库执行。
- 环境变量有示例文件。
- 依赖无明显冲突。
- 无无用文件。
- 无临时代码。

### 10.2 代码校验

- `cargo fmt` 通过。
- `cargo clippy` 通过。
- 后端测试通过。
- 前端构建通过。
- 前端关键页面可访问。
- 没有未使用依赖。
- 没有调试输出。
- 没有代码注释。

### 10.3 功能校验

必须验证：

- 注册。
- 登录。
- 邮箱验证码。
- 上传图片。
- 哈希去重。
- WebP 预览图。
- 图片列表。
- 图片详情。
- 回收站。
- 标签。
- 随机图。
- 用户组配额。
- 访客上传。
- 管理员后台。
- 存储管理。
- 审核管理。
- 迁移任务。
- 备份任务。

### 10.4 安全校验

必须验证：

- 普通用户无法访问管理员接口。
- 普通用户无法查看他人私有图片。
- 未登录用户无法访问登录后接口。
- 访客上传受开关控制。
- API Token 权限生效。
- 敏感配置不明文返回前端。
- 日志不泄露密钥。
- 上传类型限制生效。
- 上传大小限制生效。

### 10.5 移动端校验

必须验证：

- 手机能注册登录。
- 手机能上传图片。
- 手机能查看图片。
- 手机能复制链接。
- 手机能删除和恢复图片。
- 管理员手机能进入后台。
- 管理员手机能审核图片。
- 管理员手机能查看存储状态。

## 11. 交付说明规范

每次交付必须说明：

- 本次完成了什么。
- 修改了哪些文件。
- 执行了哪些测试。
- 测试结果如何。
- 是否存在未完成事项。
- 是否存在风险。
- 是否需要项目负责人确认。

禁止：

- 只说“已完成”。
- 不说明测试情况。
- 隐瞒失败测试。
- 隐瞒未完成事项。
- 隐瞒破坏性变更。

## 12. 文档维护规范

当需求变化时，必须先更新对应文档：

- 需求变化：更新需求文档。
- 数据库变化：更新数据库设计。
- 接口变化：更新接口文档。
- 页面变化：更新前端页面清单。
- 存储变化：更新存储适配器设计。
- 排期变化：更新开发排期。
- 工程规范变化：更新本文档。

代码必须服从文档。文档必须反映真实需求。

## 13. 禁止清单

严禁：

- 不看文档直接开发。
- 未确认需求就实现。
- 绕过权限。
- 绕过配额。
- 绕过审核。
- 绕过存储适配器。
- 在代码中写注释。
- 提交调试输出。
- 明文保存密钥。
- 明文保存密码。
- 上传危险文件直出。
- 只测成功路径。
- 不测移动端。
- 不测管理员权限。
- 不测失败场景。
- 交付未构建项目。
- 交付未测试功能。

## 14. 完成定义

一个功能只有同时满足以下条件，才算完成：

- 需求文档中对应功能已实现。
- 数据库变更已通过 migration 管理。
- 接口符合接口文档。
- 前端页面符合页面清单。
- 权限控制正确。
- 配额控制正确。
- 错误处理完整。
- 日志记录合理。
- 测试通过。
- 移动端可用。
- 自检完成。
- 交付说明完整。

未满足以上任一项，不能标记为完成。
