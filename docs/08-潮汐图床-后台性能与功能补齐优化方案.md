# 潮汐图床后台性能与功能补齐优化方案

## 1. 背景与目标

当前后台已经覆盖仪表盘、用户、图片、上传、配额、标签、随机图、审核、存储、外观、系统、日志、迁移和备份恢复能力。前端一级菜单统一精简为仪表盘、图片与标签、用户与配额、审核管理、存储与迁移、系统设置 6 项，避免运营入口过碎；上传、随机图、外观、日志和备份恢复收纳到系统设置。

本轮优化目标：

- 后台首屏避免拉取大列表。
- 所有大表提供分页、筛选和刷新状态。
- 仪表盘使用轻量聚合接口。
- 后台一级菜单控制在 6 项以内。
- 补齐常用后台操作：用户搜索/详情、用户图片入口、标签搜索、存储编辑与启停、任务/日志分页。
- 存储管理补齐 provider 编辑、健康检查、测试上传、测试删除和存储路由策略。
- 前端统一为融合潮汐蓝白主题和 MCP 琉璃质感的高斯模糊材料风格，并减少大面积装饰背景造成的重绘压力。
- 保持现有接口兼容，不做破坏式数据迁移。

验收基线按中大型站点设计：10 万图片、1 万用户、10 万日志/任务记录。

## 2. 后端接口与数据访问

新增兼容式聚合接口：

```text
GET /api/admin/dashboard
```

返回用户总数、今日注册、图片总数、今日上传、存储占用、待审核图片、AI 审核统计、最近上传、最近封禁和存储健康摘要。前端仪表盘只调用该接口，不再通过用户、图片、审核列表接口拼统计。

保留旧数组接口，同时新增分页接口：

```text
GET /api/admin/users/page?page=1&page_size=40&q=&role=&status=
GET /api/admin/audit/tasks/page?page=1&page_size=40&status=&q=
GET /api/admin/audit/logs/page?page=1&page_size=40&status=&q=
GET /api/admin/logs/system/page?page=1&page_size=40&q=&level=
GET /api/admin/logs/operations/page?page=1&page_size=40&q=
GET /api/admin/migrations/page?page=1&page_size=40&status=
GET /api/admin/migrations/{task_id}/items/page?page=1&page_size=80&status=
GET /api/admin/backups/page?page=1&page_size=40&status=
```

图片接口继续使用现有 `GET /api/admin/images` 的 `Page<ImageSummary>` 返回结构，前端补页码、总数和每页限制。

存储 provider 更新接口继续复用：

```text
PUT /api/admin/storage/providers/{provider_id}
```

前端可通过该接口保存名称、启用状态和配置。优先级修改保持增量兼容：若后端支持 `priority` 字段则保存，否则页面只展示不破坏原行为。

存储路由接口：

```text
GET    /api/admin/storage/routes/page?page=1&page_size=20&q=&role=&status=
GET    /api/admin/storage/routes/summary
POST   /api/admin/storage/routes
PUT    /api/admin/storage/routes/{route_id}
POST   /api/admin/storage/routes/{route_id}/enable
POST   /api/admin/storage/routes/{route_id}/disable
DELETE /api/admin/storage/routes/{route_id}
```

上传路由匹配顺序为 `user > group > role > global > quota_rule.default_storage_provider_id > default_provider`。路由只选择启用的 provider。

## 3. 前端后台改造

后台页面统一行为：

- 大表默认 `page_size` 在 20 到 80 之间。
- 显示总数、当前页、上一页/下一页、刷新按钮。
- 搜索或筛选后回到第一页。
- 加载失败只显示短错误提示，不展示原始 JSON 或 HTML。
- 详情区按需加载，避免列表首屏携带大字段。

重点页面：

- 仪表盘：切换到 `/api/admin/dashboard`，展示聚合统计、最近上传、最近封禁、存储健康摘要。
- 图片与标签：保留图片筛选、分页、总数、删除/恢复/永久删除入口；标签只作为二级能力，保留搜索、禁用和使用量查看，不作为一级菜单。
- 用户与配额：增加搜索、角色/状态筛选、用户详情、查看用户图片入口；用户组、配额规则和单用户配额覆盖归并在同一入口。
- 审核管理：任务和日志分页，详情按需加载。
- 存储与迁移：补编辑、启用/停用、配置回填、健康检查、存储路由和迁移任务，避免只能新增不能维护。
- 存储路由：提供路由表单、路由列表、分页筛选、启停、删除和存储 provider 下拉，避免手工复制 UUID。
- 系统设置：收纳站点、上传、随机图、外观、SMTP、验证码、日志、备份恢复；日志和备份继续分页加载，详情按需加载。

前端性能与体验：

- 统一半透明材料层和 8px 圆角，减少嵌套卡片。
- 表格横向滚动；移动端后台使用抽屉菜单，并在窄屏把后台表格转为卡片列表。
- 统一 loading、空状态、toast、确认弹窗、分页器、状态徽章和 reduced-motion 动效降级。
- 图片卡片启用 `content-visibility: auto` 和稳定尺寸，降低长列表渲染成本。
- API 请求保持同源相对路径，图片链接通过后端 `PUBLIC_BASE_URL` 和前端当前 origin 归一化。

## 4. 索引与性能

新增迁移补高频后台查询索引：

```sql
CREATE INDEX IF NOT EXISTS idx_images_status_created_at ON images(status, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_images_user_created_at ON images(user_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_images_guest_created_at ON images(is_guest_upload, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_audit_tasks_status_created_at ON audit_tasks(status, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_audit_results_created_at ON audit_results(created_at DESC);
CREATE INDEX IF NOT EXISTS idx_system_logs_created_at ON system_logs(created_at DESC);
CREATE INDEX IF NOT EXISTS idx_admin_operation_logs_created_at ON admin_operation_logs(created_at DESC);
CREATE INDEX IF NOT EXISTS idx_migration_tasks_status_created_at ON migration_tasks(status, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_migration_task_items_task_status_created_at ON migration_task_items(migration_task_id, status, created_at);
CREATE INDEX IF NOT EXISTS idx_backup_tasks_status_created_at ON backup_tasks(status, created_at DESC);
```

这些索引用于后台按状态、用户、时间倒序分页和统计，保持表结构不变。

## 5. 验收标准

- `GET /api/admin/dashboard` 可单次返回仪表盘所需数据。
- 旧后台数组接口仍可访问，现有 smoke 不失败。
- 新分页接口返回 `Page<Value>`，包含 `items`、`page`、`page_size`、`total`。
- 后台大表默认不再一次请求 200/500/1000 条。
- 移动端后台菜单、用户、审核、存储、迁移、备份页面仍可访问。
- 页面中不出现原始 JSON dump 或 `<pre>` 调试输出。
- `cargo test` 和前端编译检查通过，或记录明确的环境阻塞原因。
- 后台手工验收覆盖存储 provider 分页、健康检查、存储路由 CRUD 和测试上传回读。
